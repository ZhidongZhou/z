1.Merkle Trees 默克尔树

蓝色代表 Leaf Node 叶子结点, 红色代表 Internal Node 内部节点

1.a: 需要通过哪些节点证明节点 7 (Node 7)是包含在集合中的? 证明过程中需要重新计算哪些节点?

Inclusion Proof 包含证明

  • 首先证明节点 7被包含在集合中需要通过 Merkle Proof 默克尔证明: 通过节点 7自己 & 节点: \(\{\color{green}{6, d, e}\}\) 证明节点 7 是被包含在集合中的。
    1. 因为需要通过节点 \(\{\color{green}6, 7\}\) 计算出 \(\color{blue}c\)
    2. 然后需要节点 \(\{\color{blue}{c}, \color{green}d\}\) 计算出 \(\color{blue}{f}\)
    3. 最后需要通过 \(\{\color{blue}f,\color{green}e\}\) 计算出 \(\color{blue}g\)
  • 那么 \(\{\color{blue} c, \color{blue}f , \color{blue}g \}\) 则为需要重新被计算的节点
  • 最后只需要通过对比: 重新计算的节点 \(\color{blue}{g}\) & 原本的 Merkle Root 默克尔根就可以确定集合是否原本就包含节点 7

1.b: 如果验证者知道树的叶子结点是有序的, 如何证明节点 3不在这棵树提交的集合中？

Non-inclusion Proof 不包含证明

! 这道题跟没问一样, 主要就是强调一下Adjacent Node 相邻节点以及有序排列。

  • 由于节点 2 & 节点 4为Adjacent Node 相邻节点, 并且所有Leaf Node 叶子结点有序排列。 那么只需要验证 节点 2 & 节点 4 的Inclusion Proof 包含证明就等于证明了节点 3 的Non-inclusion Proof 不包含证明。

1.c: 对于一个有\(n\)节点的默克尔树, Inclusion Proof 包含证明的复杂度

Inclusion Proof 包含证明的复杂度:

  • 首先我们可以从1.a: 的解答中看到, 一个有\(8\)个叶子结点的默克尔树只需要存储除根节点以为的节点, 因为根节点可以通过最后的 \(\{\color{blue}f,\color{green}e\}\) 计算得出。

如果现在树的验证者, 在\(\log_2(k)\)层, 存储了\(k\)个内部节点, 那么需要包含证明需要的复杂度是多少? 又需要多少额外的步骤去证明?

  • 以证明节点\(2\)为例\(\log_2(k)\)层的高度就等于目前证明的复杂度, 而\(k\)为子树的叶子结点的个数(\(4\))。
  • 所以我们可以得到在以知\(\log_2(k) = \log_2(4) = 2\)层(这里指树的深度也就是子树节点\(\{e, f\}\)的位置)的前提下, 对于节点\(2\)的包含证明同样需要\(\log_2(k)=2\)。 这个\(2\)步也就是
    1. 通过节点\(\{2, \color{green}1 \}\)计算出节点\(\{\color{blue}a\}\)
    2. 通过节点\(\{\color{blue}a,\color{green} b\}\)计算出节点\(\{\color{blue}e\}\)
  • 额外的证明工作为从节点\(\{\color{blue}e\}\)到节点\(\{\color{red}{g} \}\)的距离, 同样需要经过\(\{\color{blue}e , \color{green}f\}\)的计算得出, 也就是通过\(\log_2{n} - \log_2{k} = \log_2{\frac{8}{4}} = 1\)计算出额外的步骤为\(1\)。
  • 那么额外的验证需要的时间复杂度可以通过\(\Theta(\log_2(\frac{n}{k}))\)表示, 可以看出本题中所谓的Optimization指的就是通过增加验证者的空间复杂度(存储更多的子树的根结点)来减少时间复杂度(证明步骤)。

2.Signature Schemes 签名方案

我们需要Security Parameters 安全参数 \(\lambda\), 如果攻击者可以发起 \(Polynomial(\lambda)\) 运算。 我们希望攻击者赢得Existential Forgery 存在性伪造游戏概率为 \(2^{-\lambda}\)。

2.a: 要满足以上概率, Signature Schemes 签名方案通过\(keyGen(sk, pk)\)生成的 Publick Key 公钥 (\(pk\)) 的最短长度应该是多少?

已知攻击者可以进行 \(Polynomial(\lambda)\) 运算, 即攻击者可以最多发起\(2^{\lambda}\)次运算。

那么如果一个攻击者可以发起 \(2^{\lambda}\)次运算, 他就可以生成\(2^{\lambda}\)组Key Pairs 钥匙对。

并希望其中一组Key Pairs 钥匙对是正确的。

那么我们需要多少个\(n\)才能使得\(2^{\lambda}\)组Key Pairs 钥匙对中有一个正确钥匙对的概率达到\(2^{-\lambda}\):

\(2\lambda * \frac{1}{n} = 2^{-\lambda} \\ 2\lambda\frac{1}{n}=\frac{1}{2^{\lambda}} \\ x = 2^{2\lambda} \\ = 4^{\lambda}\)

所以一组Key Pairs 钥匙对有一个正确钥匙对的概率达到\(2^{-\lambda}\), 需要\(4^{\lambda}\), 那么一个\(pk\)大小则最少需要\(2\lambda\)。

2.b: 那么最少需要签名的大小有多大?

相同的攻击者可以最多发起\(2^{\lambda}\)次运算, 那么最多可以生成&验证\(2^{\lambda}\)大小的签名。那么最多全部签名的大小则为:

\(2\lambda\frac{1}{n} = \frac{1}{2^{\lambda}} \\ x = 2^{2\lambda}\)

每个签名\(\sigma\)的大小则至少需要\(2\lambda\)

3.Birthday Paradox 生日悖论

3.a: 如果假设: 非闰年的情况, 且生日是Uniformly Distributed 均匀分布的。那么随机两个人在同一天过生日的概率是多少?

如果Alice在1月1日过生日, 那么Bob将有\(\frac{1}{365}\)的概率在同一天过生日, 所以无论Alice的生日是哪一天, 另一个人有相同生日的概率都是\(\frac{1}{365}\) (但这一切都是基于Uniformly Distrition 均匀分布的假设, 在现实生活中人们的生日不是均匀分布的)。

3.b: 一个屋子里有\(N\)个人, 从\(N\)个人中取出\(2\)个人的不同组合可以通过 \({N\choose{2}}\) 表示, 即 \(\frac{N}{2!(N-2)!}\) 也可以简化为 \({N \choose 2} = N(N-1)/2\), 那么两个人是相同生日的概率是多少?

设每个唯一的随机对儿Pair为: \({\color{orange}{p}}\), 一个随机选取的Pair(\(p\))为 \({\color{blue}{I}}_{\color{orange}{p}}\)

如果\({\color{blue}{I}}_{\color{orange}{p}} = 0\), 那么表示随机抽取的两人生日不同。

如果\({\color{blue}{I}}_{\color{orange}{p}} = 1\), 表示表示随机抽取的两人生日相同。

通过Linearity of Expectionas 线性期望得出:

\(E(\sum_{p} I_{p}) = \sum_{p}E(I_{p}) \\ = \sum_{p}E( \frac{1}{365} * 1 + \frac{364}{365}*0) \\ = {N \choose 2} \frac{1}{365}\)

我们可以通过架设总人数\(N = 100\)进行验证, \({100\choose2}\frac{1}{365} = 4950\frac{1}{365} = 13.56\), 也就是说100个人里面做随机抽取不同的2人组, 总共有4950个2人组, 这其中有大概会有13个组有2个人是相同的生日(发生碰撞)。

3.c: 那么我们现在如果想要在\(N\)个人中至少有1个随机2人组中发生"生日碰撞"(同生缘)至少需要多少个人(\(N\))?

这里我们已知期望结果为\(1\), 将\(1\)带入上述公式:

\({N \choose 2} \frac{1}{365} = 1 \\ \frac{N!}{2!(N-2)!} \frac{1}{365} = 1 \\ N=\frac{1\pm\sqrt{2921}}{2} \\ N = 27.52, N = -26.52\)

那么我们得出最小需要大改\(N = 28\)才能至少发生一次"生日碰撞"

3.d: 传统的"生日悖论"表示一个有23人出现一对同生缘的概率是\(50.7\%\)。但是3.c: 中我们计算出23个人中有出现一对同生缘的概率是\(100\%\)

首先传统"生日悖论"所有人中没有同生缘的概率: \(P = \frac{365}{365} * \frac{364}{365} \dots \frac{365-(N-1)}{365}\)

上述公式可以理解为, N个人中第1个人的生日是365天中的任意一天 \(\frac{365}{365}\), 但是第2个人只有在剩下的364天中的一天才能不产生同生缘(第2个人与第1个人生日不相同的概率是), 以此类推到第\(N\)个人的生日只能是\(365-N+1\)天中的一天。 所以简化后得到: \(\frac{365!}{365^{N} (365-N)!}\)

即首先传统"生日悖论"所有人中出现同生缘的概率为: \(1-\frac{365!}{365^{n} (365-N)!}\)

那么通过计算我们发现当$N = 23 $时: \(1-\frac{365!}{365^{23} (365-23)!} = 0.507 \approx 50\%\)

传统"生日悖论"求的是Median 中位数, 然而我们求的是Mean 中位数(期望值), 根据曲线可以看出这两个值的不同。

所以对于概率题要注意: Median 中位数和期望值的区别。

3.e: 如果想要在\(N\)个人中至少有1个随机3人组出现中同生缘至少需要多少个人(\(N\))?

与3.b: 类似, 不过现在需要 \({N \choose 3} \frac{1}{365^2} = 1\)。

\(\frac{N!}{3!\left(N-3\right)!} \frac{1}{365^2}=1 \\ =93.81 \\ \approx 94\)

4.Bitcoin Address 比特币地址

比特币地址通常通过 base56 格式的 160-bit 哈希进行交换, 如下所示:

  • 3HHZfn1wo93snzEf5kDmkSx1vPmyCxYAxm [长度只有34位]

4.a: 如果比特币地址是使用 56 个字符的字母格式化的 160 位哈希(二进制), 那么它有多少个字符？其中56个字符包括: 23456789ABCDEFGHJKLMNPQRSTUVWXYZabcdefghijkmnpqrstuvwxyz

设需要\(n\)个base56来表示需要\(56^n\)长度组合的字符串, 现在我们有\(2^{160}\)位长度组合的字符串。 那么我么只需要求解: \(56^n = 2^{160}\)得到: \(n = 27.6 \approx 28\)

即, 需要至少28个base56的字符来满足对160位长度组合的格式交换

4.b: 通过观察base56的字符, 能够发现那些字符没有被使用?

1,0,I,o,l,O

主要愿意是为了防止混淆, 发生转账地址错误的情况。

4.c: 为什么将Publick Key Hash 公钥的哈希值作为地址而不是Public Key 公钥 (\(pk\))本身进行分发是安全的？

只要哈希函数\(H()\)是Collision Resistant 防碰撞的, 那么将十分困难生成一个公钥\(pk\), 使其哈希结果与目标哈希结果相同的情况。

因为不会发生碰撞, 所以分发公钥的哈希值\(H(pk)\)相对安全。

4.d: 如果Bob的比特币地址\(X\)是通过对其公钥的哈希加密得到的\(\color{green}X=H(pk_{Bob})\). 此时攻击者Mallory找到了一个Collision 碰撞(即生成一个\(PK_{mallory}\)使得\(\color{red}{X}=H(pk_{Mallory})\)). 那么这将造成什么影响?

如果在Public Key Hash 公钥哈希下收到的比特币被锁定在标准 P2PKH 脚本下。 那么 Mallory 可以通过 Bob 的钱包使用对应的比特币, 因为Mallory可以通过P2PKH中ScriptPubKey脚本中OP_EQUALVERIFY的验证。

4.e: 使用Birthday Attack 生日攻击, 对于哈希函数\(H()\)输出长度为\(t\)的结果, 仅需\(O(2^{t/2})\)位的难度以找到Collision 碰撞。 生日攻击对4.d: 适用吗？为什么或者为什么不？如果不是, Mallory 必须期望花费多少工作才能找到这样的Publick Key 公钥？

不适用: 因为4.d: 中生日攻击被用来在已知输入中找到一对儿或一组输入找到Collision 碰撞。 但是对于 Mallory 来说, 她现在需要找到一个输入: 公钥(\(\color{red}{pk_{Mallory}}\))通过\(H({\color{red}pk_{Mallory}})\)后与一个已知的输出Public Key Hash 公钥哈希(\(\color{green}X\))发生碰撞。

但是如果 Mallory 知道Bob的公钥 \(\color{green}pk_{Bob}\), 那么问题就将从破解原像防御(Pre-Image resistance), 变为破解弱碰撞防御(Second pre-image resistance):

  1. 原像防御(Pre-Image resistance): 就是任意给一个hash值\({\color{green}{X}}\), 你很难破解出它的原像\({\color{green}{pk_{Bob}}}\), 使得\(H({\color{red}{pk_{Mallory}}})\)=(\({\color{green}{X}}\)), 这个难度一般是NP难度的, 也就是所谓的单向函数。
  2. 弱碰撞防御(Second pre-image resistance): 就是给定一个\(\color{green}{pk_{Bob}}\), 你很难找到另一个\(H({\color{red}{pk_{Mallory}}})\), 使得 \(H({\color{red}{pk_{Mallory}}}) = H({\color{green}{pk_{Bob}}})\)。
  3. 强碰撞防御(Collision resistance): 就是你很难找到两个不同的\({\color{green}{pk_{Bob}}}\) 和 \({\color{red}{pk_{Mallory}}}\), 使得\(H({\color{red}{pk_{Mallory}}})=H({\color{green}{pk_{Bob}}})\)。

如果符合强碰撞防御, 也就是会符合弱碰撞防御, 但是不一定符合原像防御。

如果只满1,2个, 在密码学上是不安全的, 一般加密哈希函数应该满足1, 2, 3。

那么相比: \(H({\color{red}{pk_{Mallory}}})=({\color{green}{X}}\)), 破解 \({H(\color{red}{pk_{Mallory}}}) = H({\color{green}{pk_{Bob}}})\)相对容易。

所以如果 Mallory 通过一些途径(比如Bob之前用\(\color{green}{pk_{Bob}}\)签署了某个交易)得知了\(\color{green}{pk_{Bob}}\), 那么 Mallory 需要尝试所有长度为\(t\)的二进制字符串 \(2^t\) 个不同的\(\color{red}{pk}\)'s并期望其中一个\(\color{red}{pk}\)的结果与\(\color{green}{X}\)发生碰撞。

5.UTXO-based Transaction 基于未花费的交易输出的交易

Tx Inputs Outputs
1 ∅ \({\color{red}{25 → A}}\); \({\color{red}{25 → B}}\)
2 1[0] \(10 → A\); \({\color{red}{13 → C}}\); \(1 → D\)
3 2[1] \(2 → A\); \(2 → B\); \({\color{red}{4 → C}}\); \(2 → D\); \({\color{red}{2 → E}}\)
4 1[1];3[4] \(4 → A\); \(11 → B\); \({\color{red}{1 → C}}\); \(5 → D\); \(5 → E\)
5 4[2];3[2] \(1 → C\); \(1 → D\); \(2 → E\)

5.a: 在交易Tx-5之后, \({A,B,C,E,E}\) 5个键, 各控制多少钱?

排除掉在其他交易中被再次作为Inputs的\(\color{red}{Outputs}\)

\(\{A:16, B:13, C:1, D:9, E:7\}\):

  • \(A = 0+10+2+4 = 16\)

  • \(B = 0+2+11=13\)

  • \(C = 0 +1 = 1\)

  • \(D = 0 + 1+2+5+1=9\)

  • \(E = 5+2=7\)

5.b: 交易Tx-5之后的 UTXO 集合的状态是什么？

排除掉在其他交易中被再次作为Inputs的\(\color{red}{Outputs}\), 剩下的未花费的交易输出(UTXO):

\(\{2[0], 2[1], 3[0], 3[1], 3[3], 4[0], 4[1], 4[3], 4[4], 5[0], 5[1], 5[2]\}\)

5.c: 从交易Tx-1 到 Tx-5, 所花费的交易手续费是多少？

合计所有未花费的交易输出(UTXO): \(10+1+2+2+2+4+11+5+5+1+1+2 = 46\)

用总输入 \(\{A:25 + B:25\} = 50\) 减去未花费(UTXO)的总和得到: \(50 - 46 = 4\)

所以交易费为\(4\)
